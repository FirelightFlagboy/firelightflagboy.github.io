This is the recap of a live coding during a meeting with a company.
I was in front of a jury of 3 people.

The goal of this test was more for the jury to know how I work on a specific feature/project, rather than creating an optimized code in live.

Assessment

Write a function in Java that take 1 integer argument N and return 1 integer argument. The returned integer is the sum of N prime number.

func AddNPrime(int n) int {}

What is a prime number

A prime number is a number that can only be divided by 2 numbers: 1 and itself.

Corollary

Since a prime number must be divided by 2 numbers: 1 is not a prime number.

List of first prime number
2 3 5
7 11 13
17 19 23

Create IsPrime

To begin the assessment, I decided to create a utility method to know if a number is prime. I decided to create the class Nprime and name the method isPrime (pretty straight forward).

public class Nprime {
  public static boolean isPrime(int n) {
    for (int i = 2; i < n; ++i) {
      if (n % i == 0)
        return false;
    }
    return true;
  }
}

At this step, isPrime doesn’t respect the corollary for a prime number. This is resolve in a later in step.

Now that I have create the method, I need to test it.

I asked which function to use for an assertion.
They suggest me a simple System.out.println or assert.
I choose assert: a wrong choice I made.

public class Main {
  public static void testIsPrime() {
    System.out.println("test is prime");
    assert(Nprime.isPrime(1) == false);
    assert(Nprime.isPrime(2) == true);
    assert(Nprime.isPrime(5) == true);
    assert(Nprime.isPrime(9) == false);
  }

  public static void main(String[] args) {
    testIsPrime();
    System.out.println("good");
  }
}

So, by executing my test, I see no error

Create NextPrime

With the utility function isPrime, I need another one to determine the next prime number. I decided the meaningful name nextPrime.

public class Nprime {
  ...
  public static int nextPrime(int n) {
    while (isPrime(++n) == false)
      ;
    return n;
  }
}

With the method done, I create more test to ensure the quality of my code.

Create AddNPrime

Now I have all the method needed to create the main function of the assessment: addNPrime.

public class Nprime {
  ...
  public static int addNPrime(int n) {
    int prime = 2;
    int sum = 0;

    while (n > 0) {
      prime = nextPrime(prime);
      sum += prime;
      n--;
    }
    return sum;
  }
}

New Demand

At this step, I finish the assessment.
The jury create a new task:

Improve the method addNPrime() to receive another integer argument which will be a step, e.g.: with N = 2 and step = 1 as argument: it will:

  1. count 2
  2. skip 3
  3. count 5
  4. returning 7

Method Overload

For this new task, I decided It is more interesting to do a method overload than just rewriting the method and test.

public class Nprime {
  ...
  public static int addNPrime(int n) {
    return addNPrime(n, 0);
  }

  public static int addNPrime(int n, int step) {
    int prime = 2;
    int sum = 0;

    while (n > 0) {
      int i = 0;
      do {
        prime = nextPrime(prime);
        i++;
      } while (i < step);
      sum += prime;
      n--;
    }
    return sum;
  }
}

Who say new method, say new test suite.

public class Main {

    assert(Nprime.addNPrime(3, 1), 18);
    assert(Nprime.addNPrime(4, 1), 31);

  public static void main(String[] args) {

Something is wrong

When testing Nprime.addPrime(4, 1) the result should be 35 (2 + 5 + 11 + 17) not 31
But assert did not raise any exception (I was executing without -ea to enable assertion) because assert was disabled.

public class Nprime {

      if (n % i == 0)
        return false;
    }
    return true;

Feedback

Refactoring the Test Output

I did not like how the main generate a lot of output. I decided to refactor the Main class to use a test() method.

public class Main {
  public static void test(Object got, Object expected) {
    if (got != expected)
      System.out.println(String.format("got %s, expected %s", got, expected));
    else
      System.out.print(".");
  }

  public static void testIsPrime() {
    System.out.print("test is prime ");
    test(Nprime.isPrime(1), false);
    test(Nprime.isPrime(2), true);
    test(Nprime.isPrime(5), true);
    test(Nprime.isPrime(9), false);
    System.out.println("");
  }
  ...
}

Correct AddNPrime error

the error in addNPrime was caused because:

  1. prime was initialized at 2, so 2 was always skipped.
  2. I skiped the number before doing the addition.
public class Nprime {
  ...
  public static int addNPrime(int n, int step) {
    int prime = 1;
    int sum = 0;

    while (n > 0) {
      prime = nextPrime(prime);
      sum += prime;
      prime = skipPrime(prime, step);
      n--;
    }
    return sum;
  }

  private static int skipPrime(int prime, int step) {
    while (step > 0) {
      prime = nextPrime(prime);
      --step;
    }
    return prime;
  }
}

This time, we do not have any error

$ java Main
test is prime ....
test is next prime ....
test is add n prime ....
test is add n prime step ....

Create a new class

From the feedback, I segregate Nprime by creating a new class that contain only the method related to math.

public class PrimeMath {
  public static int addNPrime(int n) { ... }

  public static int addNPrime(int n, int step) { ... }

  private static int skipPrime(int prime, int step) { ... }
}

Now Nprime is a lot clearer and respect the SRP

public class Nprime {
  public static boolean isPrime(int n) { ... }

  public static int nextPrime(int n) { ... }
}

Conclusion

The assessment although simple, can be quite tedious if you do not have organized thought: a plan.
Even a simple exercise/feature can be push way further.

When developing, it’s better to stick with something tried’n’true than something you think know how it works: see how assert betrayed me.

I think the live coding sessions would have been easier, if I was able to use a testing framework like Junit.
